5. **设置线程的属性** -- lw

   **一、设置线程属性核心思路**

   1. 先定义属性变量：`pthread_attr_t attr;`
   2. 初始化：`pthread_attr_init(&attr);`
   3. 设置各种属性（detach、stack、sched 等）
   4. 创建线程时传入 `&attr`
   5. 销毁属性：`pthread_attr_destroy(&attr);`

   ------

   **二、常用属性（按使用频率排序）**

   **1. 分离状态（最常用）**

   控制线程是否需要 `pthread_join` 等待回收。

   ```c
   // 设置为分离线程（自动释放，无需 join）
   pthread_attr_setdetachstate(&attr, PTHREAD_CREATE_DETACHED);
   
   // 默认：可接合线程（需要 join）
   pthread_attr_setdetachstate(&attr, PTHREAD_CREATE_JOINABLE);
   ```

   **2. 线程栈大小**

   防止递归 / 大数组导致栈溢出。

   ```c
   size_t stack_size = 1024 * 1024 * 8;  // 8MB
   pthread_attr_setstacksize(&attr, stack_size);
   ```

   **3. 调度策略（实时线程用）**

   - `SCHED_OTHER`：默认普通调度
   - `SCHED_FIFO`：实时先进先出
   - `SCHED_RR`：实时轮转

   ```c
   pthread_attr_setschedpolicy(&attr, SCHED_RR);
   ```

   **4. 线程优先级**

   配合实时调度使用，需要 root。

   ```c
   struct sched_param param;
   param.sched_priority = 10;  // 1~99，越大优先级越高
   pthread_attr_setschedparam(&attr, &param);
   ```

   **5. 继承调度策略**

   默认是继承创建者的优先级，想自己设置要关掉：

   ```c
   pthread_attr_setinheritsched(&attr, PTHREAD_EXPLICIT_SCHED);
   ```

   **6. 栈警戒区大小（防溢出）**

   ```c
   size_t guard_size = 1024 * 4;
   pthread_attr_setguardsize(&attr, guard_size);
   ```

   

6. **UDP面向应用报文传输与TCP面向字节流传输的疑惑**？ -- lzw

#### 报文本质也是数据。

### 1.为什么UDP面向应用报文就不可靠呢？

1）**不确认是否收到**

发出去就不管了，对方收没收到 UDP 根本不知道。

（UCP去厕所，他只管拉，拉外面还是坑里，他不知道）

2）**丢包不重传**

网络一拥堵、一抖动，报文直接丢，UDP 不会补发。

（已经拉完了，不会因为没拉进坑就重新再拉一次）

3）**不保证顺序**

先发的报文可能后到，UDP 不会重新排序。

（站得高拉的远，一开始拉小的，后面拉大的，大的拉得晚，但是落得快）

4）**不做流量控制 / 拥塞控制**

你疯狂发，网络扛不住就疯狂丢。

（拉得太多，坑都爆炸🌶️！我直接TM的往厕所外面流！！）

5）**没有连接状态**

发之前不握手，发完不断开，谁都能发，谁都能收。

（不挑坑，进厕所就拉，沟子不夹紧，屎就不会停）

​	**简单的来说，UDP只管把 “应用报文” 发出去，不管它到没到、乱没乱、丢没丢，讲究一个随地大小拉，有缘者得之，所以叫不可靠。**



### 2.报文本质不就是数据流吗，同样都是数据的输出问什么UDP就是为不可靠？

​	这个问题的关键不在**数据本身**，而在**协议怎么看待==边界、怎么处理丢失==**

1）**UDP 把一次应用数据当成一个不可分割的整体，丢就全丢，不补；**

2）**TCP 把所有数据当成一串连续字节，丢一点补一点，能拼回去。**

​	不管你是UDP还是TCP本质都是010010010...这种字节流。



### 3.那我接收端怎么知道你的数据顺序是如何的？你TCP凭啥字节打乱了还能输出正确的顺序？

#### TCP:字节流是 “编号” 传的，顺序早就定好了

1. 应用给 TCP 一堆数据，TCP 看成**一长串连续字节**
2. 它给**每个字节都编上序号**
3. 网络上可以随便拆成小段发
4. 到达时可能乱序、可能丢包
5. 接收端**按序号重新排队**，缺了就要求重传
6. 最后拼成完整、有序的一串，交给应用

#### UDP：报文没有全局编号，彼此独立

​	1.每个应用报文就是一个独立 UDP 包

​	2.报文之间**没有序号关联**

​	3.先发的可能后到，丢了就没了

​	4.接收端只能按到达顺序给应用，不会重排、不会补

可以总结为:

**TCP 给字节编号 → 能排序、能重传 → 可靠**

**UDP 不给报文编号 → 丢了不知道、乱了不会排 → 不可靠**
